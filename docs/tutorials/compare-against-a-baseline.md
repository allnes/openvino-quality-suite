# Compare against a baseline

Use a baseline when you want the report bundle to explain what changed, not only
what the current run measured.

## Build a baseline-aware bundle

Pass a baseline report to the report bundle command:

```bash
oviq report build \
  --report reports/current.json \
  --baseline reports/baseline.json \
  --gates configs/gates/default_gates.yaml \
  --out reports/current-bundle
```

Unknown metrics remain `unknown`; they must not be converted to pass.

## Write only the analysis JSON

If a CI job only needs machine-readable findings, use `report analyze`:

```bash
oviq report analyze \
  --report reports/current.json \
  --baseline reports/baseline.json \
  --gates configs/gates/default_gates.yaml \
  --out reports/current.analysis.json
```

The analysis file contains normalized metric observations, findings, and
comparison data when a baseline or gate file is provided.

## Review the result

Check these files first:

- `analysis.json`: finding count, metric count and comparison payload.
- `metrics.csv`: flat scalar paths for quick review.
- `index.md`: pull-request friendly summary.
- `dashboard.html`: detailed visual review.

## Baseline rules

Use a baseline generated by the same suite and comparable configuration. If a
metric cannot be computed in either report, keep that row as `unknown`; it is
better to expose missing evidence than to fabricate a pass.
